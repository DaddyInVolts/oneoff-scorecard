Section 1

Problem & Opportunity

Problem Statement

Expedite runs experiment scorecards on a fixed cadence tied to the rollout's calendar. When an experiment owner needs to look at scorecard metrics over a window that doesn't match the cadence — the three days around an outage, the last seven days of a tenant canary, a "what did this look like before the hotfix" comparison — they have no self-service path. The work falls back to filing a DS partner ticket, hand-rolling SQL, or asking the platform team to kick off a one-off run. The result is that custom-window analysis happens too late to influence the ship decision, gets done inconsistently across owners, and quietly burns DS partner cycles on operational work.

Opportunity

A "Request One-off Scorecard" action, surfaced directly inside the rollout view, lets an experiment owner pick an existing scorecard, an assignment unit, and an arbitrary UTC time window — and queue that scorecard against the same data and definitions the platform already runs on cadence. The request joins the rollout's existing scorecard table as a "Requested" row and, on completion, becomes a clickable scorecard alongside the cadence ones. Custom-window questions get answered in hours instead of days; the DS partner gets pulled in for judgment instead of operations; and every one-off becomes a first-class artifact attached to the rollout rather than a one-time email thread.

Evidence

Quantitative evidence pending. Recommended additions before review: (1) volume of ad-hoc one-off scorecard requests routed to DS partners per quarter; (2) median time from request to delivered scorecard today; (3) rate of ship decisions that wait on, or skip, a needed custom-window scorecard.
  • Anecdotal — DS partners: recurring ask-by-email pattern for "run scorecard X over window Y" with no shared queue or self-service surface.
  • Anecdotal — Experiment owners: cadence scorecard windows often miss launch, hotfix, or incident windows where the decision-relevant signal actually lives.
  • Adjacent signal: the rollout-level scorecard table already renders Completed scorecards consistently — the data path and UI surface for a "Requested → Completed" extension are in place.
Section 2

Goals & Success Metrics

Goals

  1. Give experiment owners a self-service path to request a scorecard over any UTC window without filing a ticket or writing SQL.
  2. Make the request a first-class artifact on the rollout — visible, attributable, and reusable — by surfacing it as a "Requested" row in the rollout's scorecard table alongside cadence scorecards.
  3. Reuse the existing scorecard catalog and compute pipeline — no new scorecard definitions, no parallel data path — so cadence and one-off results are directly comparable.

Success Metrics

Primary Metric
Self-service one-off scorecards per quarter
Target: 50+ per quarter by Q+2 · Baseline: ~0 (no self-service path today)
Secondary Metric
DS-routed scorecard requests per quarter
Target: 50% reduction vs. pre-launch baseline — the one-off feature absorbs the ad-hoc volume
Guardrail Metric
Median time from Requested → Completed
Must not exceed TBD (target ≤ 4 hours); regression indicates the compute pipeline is saturated by self-service load

Explicit Non-Goals

Authoring new scorecard definitions — the request flow only selects from the curated catalog
Real-time or streaming scorecards — one-off runs are batch over a closed window
Cross-experiment or cross-rollout aggregation — each request targets a single rollout
Custom metric or filter authoring inside the drawer — out of scope for v1
Section 3

User Scenarios

Scenario 1 — Experiment owner running a launch retrospective (Primary)

Priya is the owner of a Bizchat feature that ran a 30-day A/B. On days 12–14 a regional outage skewed user behavior. The default cadence scorecard window includes those days, contaminating her retention metrics. Priya opens the rollout in Expedite, expands the Roll-out type panel, clicks Request One-off Scorecard, picks "Bizchat Indicator Scorecard" by User Id, sets the window to day 15 → day 30 in UTC, and clicks Validate. A "Requested" row appears in her scorecard table. Hours later it flips to "Completed" and links to the same scorecard surface she already knows — clean numbers, no DS ticket.

Scenario 2 — DS partner triaging a guardrail regression

Raj is a DS partner. The weekly Windows Guardrails — Commercial scorecard flagged a regression and he wants to confirm by tightening the window to the suspected three-day incident. He opens the rollout, picks "Windows Guardrails — Commercial" by Device Id, sets a 3-day window, validates, and gets a clean comparison run without writing SQL or touching the cadence config. The Requested row carries his alias and timestamp, so the team has a durable record of what was investigated and why.

Scenario 3 — PM doing a launch readiness check before ring expansion

Marcus is a PM about to expand a Copilot feature from canary to Tier 1. He wants a tenant-level engagement read over the last 14 days of canary, but the cadence scorecard runs at User Id. He opens the rollout, requests "Commercial Copilot Engagement" by Tenant Id over the 14-day window, validates, and gets the tenant view he needs before the expansion decision — no parallel ask to DS, no last-minute analyst pull, and the request is attached to the rollout for the reviewer to inspect.

Section 5

Design Direction

UX Flow & Reference

Existing rollout panel in the Expedite experiment workspace — design system, fonts, and Fluent UI patterns to be retained.

Rollout panel (top → bottom):
Collapsible header "Roll-out type" → Experiment link (with OCG / Control / Treatment flight IDs) → Primary actions row ("View configuration" + "Request One-off Scorecard") → Scorecard table (Name · Period · Status)

Request drawer (opens on "Request One-off Scorecard"):
Header "Scorecard Options" → Assignment Unit dropdown → Scorecard dropdown → Analysis time range block (Start date + Start time, End date + End time, with explicit "timezone: GMT-0000 (UTC)" label and info tooltip) → Helper text "This will create one scorecard using the data within the time window provided" → Analysis Details (computed Name) → Footer: Validate button

See the live clickable prototype linked at the top of this spec.

Key Interaction Notes

  • Drawer overlay: the request drawer slides in over the rollout panel rather than navigating away — the requester keeps full context on which rollout they're acting against.
  • Status badges: reuse the existing system in the scorecard table — green "Completed" (Checkmark icon) and blue "Requested" (Clock icon). New requests inherit "Requested" immediately on Validate.
  • UTC everywhere: date and time pickers are in UTC; the timezone label is always visible — never inferred from the user's locale and never hidden behind a setting.
  • 30-minute interval times: the time dropdown uses 48 fixed 30-min slots for both Start and End, matching the platform's other scheduling controls.
  • Computed Name: shown live in the drawer but not editable in v1 — keeps the downstream parsing format stable and prevents free-form naming sprawl.

Edge Cases & Empty States

  • No prior scorecards on the rollout: the table renders with column headers and an empty body; the One-off CTA still shows and is the primary path to populate it.
  • Cap reached: Validate click surfaces a blocking inline message naming the cap and listing the requester's existing in-flight requests with submit timestamps so the user can cancel or wait.
  • Window exceeds cap: inline error on the End field naming the cap and the actual window length; submit blocked.
  • Compute pipeline unavailable: Validate succeeds optimistically; row inserts with "Requested" status; backend retries until success or timeout, at which point row flips to a third status "Failed" with a Retry action (P1 follow-up).
  • Catalog scorecard renamed mid-flight: completed rows retain the historical name as displayed at request time; new requests use the current catalog name.

Accessibility

  • Drawer is keyboard-traversable; tab order matches visual flow: Assignment Unit → Scorecard → Start date → Start time → End date → End time → Validate.
  • All date and time pickers announce label, current value, and constraint state (e.g., "End date — required, must be after Start date").
  • Status badges convey state both visually (color + icon) and textually for screen readers — "Requested" and "Completed" are read aloud as the row gains focus.
  • Helper text and inline validation messages are programmatically associated with their fields so assistive tech announces them on focus and on error.
Section 6

Open Questions & Risks

Open Questions

Q1: What is the maximum allowable one-off window?

Compute cost grows with window size; longer windows are more useful for retrospectives. Need a defensible cap before launch. Owner: Compute Platform + PM.

Q2: Who curates the scorecard catalog and how do new scorecards enter it?

Today the selectable list is effectively hardcoded. Productizing a self-service surface requires a catalog source of truth, an owner, and an onboarding path for new scorecards. Owner: Data Science + PM.

Q3: What SLA do we publish for one-off completion, and what is the escalation path when missed?

Owners will plan around the SLA. Need a documented target (e.g., median ≤ 4h, p95 ≤ 24h) and a runbook for when the queue depth is degrading. Owner: Compute Platform.

Q4: What concurrency cap per requester per rollout protects the pipeline without blocking real work?

R-13 sets a default of 3 pending the data. Need a number grounded in pipeline capacity and observed request bursts. Owner: Compute Platform + PM.

Q5: Does surfacing Tenant Id and Device Id assignment options to self-service trigger any PII / privacy review?

Cadence runs use these units today, but moving them into a self-service drawer changes the access pattern. Confirm whether any additional gating or audit is required. Owner: PM + Privacy.

Risks

Risk 1: Self-service load spikes saturate the scorecard compute pipeline

Likelihood: Medium · Impact: High. Easy self-service can flip a workflow that was rate-limited by ticket overhead into one that's only rate-limited by the user. Mitigation: per-requester concurrency cap (R-13); window-length cap (R-05); observability on the request queue with paging on depth threshold; ability to throttle the feature back to ticket-routed if needed.

Risk 2: Owners pick the wrong Assignment Unit and act on misleading numbers

Likelihood: Medium · Impact: High. A Device-Id run on a Tenant-Id-randomized experiment can produce coherent-looking but wrong results. Mitigation: pre-fill from the rollout's primary unit (R-10); inline help text on the field; per-completed-row tooltip showing the unit so reviewers can sanity-check before acting.

Risk 3: Duplicate requests proliferate because owners can't see what's already in flight

Likelihood: Medium · Impact: Medium. Multiple owners on the same rollout independently request the same scorecard/window. Mitigation: the rollout-level scorecard table is the dedup surface — Requested rows are visible to all owners; concurrency cap surfaces existing requests when triggered; P2 duplicate-aware Validate that warns on near-identical pending requests.

Risk 4: One-off and cadence definitions drift if compute pipelines fork

Likelihood: Low · Impact: High. If "one-off" gets its own compute path, the comparability promise breaks and reviewers lose trust in side-by-side reads. Mitigation: R-09 mandates a single pipeline; integration tests assert one-off vs. cadence parity on a canary scorecard; this is a launch-blocking invariant.

Risk 5: "Requested" rows accumulate stuck without status transition

Likelihood: Medium · Impact: Medium. Failed backend runs leave the row in "Requested" indefinitely; users lose trust in the surface. Mitigation: visible age on Requested rows; auto-fail-and-retry policy on the backend; "Failed" status with Retry action as a P1 follow-up; quarterly audit of stuck rows.
